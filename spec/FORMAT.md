# Cairn Format Specification

This document defines the complete format for all three layers of the `.cairn/` directory.
It is the authoritative reference for writing, validating, and tooling Cairn files.

---

## Directory Structure

Every project using Cairn places a `.cairn/` directory at its repository root:

```
.cairn/
├── output.md          # Layer 1: global constraints, always injected
├── SKILL.md           # Operating protocol; copied from skills/claude-code/SKILL.md at init
├── domains/           # Layer 2: domain design context, injected on demand
│   ├── api-layer.md
│   ├── auth.md
│   └── state-management.md
└── history/           # Layer 3: raw decision events, queried on demand
    ├── 2023-03_state-mgmt-transition.md
    ├── 2023-09_trpc-experiment-rejection.md
    └── 2024-01_auth-debt-accepted.md
```

---

## Layer 1: `output.md` — Global Constraints

### Purpose

`output.md` is injected at the start of every AI session. It establishes the constraint
framework within which all AI suggestions must operate.

### Injection Timing

**Every session, always.** The AI MUST read `.cairn/output.md` before any response.

### Token Budget

| Target | Hard Limit |
|--------|------------|
| 500 tokens | 800 tokens |

If `output.md` exceeds 800 tokens, content SHOULD be moved to `domains/` or `history/`.
A file between 500–800 tokens is acceptable. Below 500 is ideal.

### Required Sections

`output.md` MUST contain these six sections, in this order:

```markdown
## stage
## no-go
## hooks
## stack
## debt
## open questions
```

#### `## stage`

Project phase and reasoning mode.

```
## stage

phase: <name> (<YYYY-MM>+)
mode: <priority-1> > <priority-2> > <priority-3>
team: <size>, <constraint>
reject-if: <condition>
```

| Field | Required | Format | Example |
|-------|----------|--------|---------|
| `phase:` | MUST | `name (YYYY-MM+)` | `early-growth (2024-09+)` |
| `mode:` | MUST | priorities separated by `>` | `stability > speed > elegance` |
| `team:` | SHOULD | size + constraint | `2, no-ops` |
| `reject-if:` | SHOULD | plain condition | `migration > 1 week` |

#### `## no-go`

Technology directions the AI MUST NOT suggest.

```
## no-go

- <direction> (<one-line hint>)
- <direction> (<one-line hint>, see domains/<domain>.md)
```

Rules:
- Each entry is a single bullet line
- Parenthetical is a hint only — full rationale belongs in `domains/` or `history/`
- MAY be empty if no directions have been excluded yet

#### `## hooks`

Keyword-to-domain mappings that trigger domain file injection.

```
## hooks

planning / designing / suggesting for:

- <keyword list> → read domains/<domain>.md first
- <keyword list> → read domains/<domain>.md first
```

Rules:
- One line per domain
- Keywords are comma- or slash-separated
- SHOULD cover all domains listed in the project's domain list
- MAY be auto-generated by `cairn-init.sh` based on selected domains

#### `## stack`

Active technology choices. `key: value` pairs only.

```
## stack

<layer>: <technology>
<layer>: <technology>
```

Examples: `state: Zustand`, `api: REST`, `db: PostgreSQL`, `auth: JWT + Refresh`, `deploy: Railway`

#### `## debt`

Accepted technical debts that the AI MUST NOT attempt to fix.

```
## debt

<ID>: accepted | <revisit_when condition> | <constraint>
```

| Part | Required | Example |
|------|----------|---------|
| `<ID>` | MUST | `AUTH-COUPLING` |
| `accepted` | MUST | literal string |
| `revisit_when condition` | MUST | `fix when team>4 or MAU>100k` |
| `<constraint>` | MUST | `no refactor now` |

MAY be empty if no debt has been formally accepted.

#### `## open questions`

Global unresolved questions that affect future AI suggestions across the project.

```
## open questions

- <global question that has not been decided yet>
```

Rules:
- For **global** unresolved issues only — not per-domain questions (those go in `domains/`)
- NOT a no-go, NOT a debt list, NOT a design proposal backlog
- An entry that gets resolved MUST be removed; a resolved question belongs in `history/`
- MAY be empty

### Writing Rules

1. **Constraints only, no explanations.** Parentheticals are hints. Full rationale goes in `domains/` and `history/`.
2. **Structure over prose.** Use `key: value` pairs and short lists. No complete sentences.
3. **Every line MUST change AI behavior.** Remove a line and ask: would the AI's response differ? If not, delete it.
4. **Enforce the token budget.** Content above 800 tokens belongs in a deeper layer.

### Complete Example

```markdown
## stage

phase: early-growth (2024-09+)
mode: stability > speed > elegance
team: 2, no-ops
reject-if: migration > 1 week

## no-go

- tRPC (REST integration cost, see domains/api-layer.md)
- Redux (boilerplate overhead at team-2)
- kubernetes (no ops capacity)
- microservices (no B2B product)

## hooks

planning / designing / suggesting for:

- api / endpoint / tRPC / GraphQL → read domains/api-layer.md first
- auth / login / JWT / session → read domains/auth.md first
- state / store / Zustand → read domains/state-management.md first
- db / migration / ORM → read domains/database.md first

## stack

state: Zustand
api: REST
db: PostgreSQL
auth: JWT + Refresh
deploy: Railway

## debt

AUTH-COUPLING: accepted | fix when team>4 or MAU>100k | no refactor now
WS-CONCURRENCY: accepted | CDN migration resolves | no polling fallback

## open questions

- Whether billing should remain in the monolith for the next 2 quarters
- Whether auth session storage should be unified this cycle
```

## Layer 2: `domains/*.md` — Domain Design Context

### Purpose

Domain files provide pre-compressed design context for a specific area of the codebase.
They are the critical middle layer between the always-on global constraints (`output.md`)
and the unlimited raw history (`history/`).

### Injection Timing

During planning, architecture design, technology selection, or migration evaluation.
The AI SHOULD read the corresponding domain file when the user's request matches a
keyword in `output.md`'s `## hooks` section.

### Token Budget

| Per File |
|----------|
| 200–400 tokens |

### Frontmatter

Domain files MUST include YAML frontmatter before the `# <domain-name>` title heading:

```yaml
---
domain: <domain-key>
hooks: ["keyword1", "keyword2", "..."]
updated: <YYYY-MM>
status: <active | stable>
related: ["domain-name"]   # optional
---
```

| Field | Required | Format | Purpose |
|-------|----------|--------|---------|
| `domain` | MUST | kebab-case | Must match the filename stem and the `# title` heading |
| `hooks` | MUST | JSON array of strings | Keywords that trigger injection. Referenced by `output.md` hooks section. Used by Phase 3 MCP Server for precise matching without AI inference. |
| `updated` | MUST | `YYYY-MM` | Date of last substantive update. Used by `cairn status` for stale detection. |
| `status` | SHOULD | `active` or `stable` | `active` — design is evolving; `stable` — design is settled. Staleness is always computed from `updated` vs. history dates; it is never declared manually. |
| `related` | OPTIONAL | YAML flow-style string array | Declares other domain names related to this domain. MCP `cairn_match` uses this field to recommend loading the `## trajectory` section of related domains alongside the primary match. Expansion rules: BFS 1-hop only (no transitive expansion); max 2 domains expanded; author-declared order (first 2 taken); references to non-existent domains are silently dropped with a warning returned; circular references are forbidden (the primary domain itself is never expanded). |

### Required Sections

```markdown
# <domain-name>

## current design
## trajectory
## rejected paths
## known pitfalls
## open questions
```

#### `# <domain-name>`

File title. MUST match the domain key (e.g., `# api-layer`).

#### `## current design`

One to three sentences describing the current state. MUST include:
- The primary design choice in use
- Any unresolved boundary or constraint worth knowing

#### `## trajectory`

Chronological list of design changes. One line per event.

```
## trajectory

[YYYY-MM] <initial state>
[YYYY-MM] <change> → <one-line reason>
[YYYY-MM] <current state>
```

#### `## rejected paths`

Directions that were evaluated and excluded.

```
## rejected paths

- <option>: <rejection reason, one sentence>
  Re-evaluate when: <condition under which this direction is worth reconsidering>
```

The `Re-evaluate when:` note is REQUIRED for each rejected path entry.

#### `## known pitfalls`

Operational traps that the AI MUST actively consider when working in this domain.

```
## known pitfalls

- <pitfall name>: <trigger condition> / <why it happens> / <what NOT to do>
```

These are NOT debts and NOT rejections. They are persistent warnings about working
in this domain.

#### `## open questions`

Unresolved design decisions that affect future work in this domain.

```
## open questions

- <question that has not been decided yet>
```

MAY be empty.

#### `## residue checklist` (optional)

Migration cleanup obligations for this domain. Only added after a significant
migration or architectural change that requires explicit cleanup tracking.

```
## residue checklist

- [ ] <cleanup item — what must be removed or updated>
- [x] <completed item>
```

Rules:
- OPTIONAL — only add when a migration creates explicit cleanup debt
- Items are checkboxes; completed items MAY be left as `[x]` for historical record
- When all items are checked, SHOULD be removed from the domain file
- Tracked across sessions via a corresponding file in `.cairn/audits/`

### Lifecycle

- **Create:** After 2–3 `history/` entries accumulate for a domain. Generate from
  history entries using AI, confirm with a human. Do NOT write by hand before history exists.
- **Update:** Overwrite the entire file (never append). The file reflects the current
  state only. Raw events remain in `history/`.
- **Length:** Files MUST NOT grow over time. Content is replaced, not accumulated.

### Complete Example

```markdown
---
domain: api-layer
hooks: ["api", "endpoint", "tRPC", "GraphQL", "REST", "OpenAPI"]
updated: 2024-03
status: stable
related: ["auth"]
---

# api-layer

## current design

REST + OpenAPI. No GraphQL, no tRPC. All endpoints follow /v1/ prefix,
but versioning strategy is not formally defined. Error format partially
inconsistent — new endpoints MUST use { code, message, data } structure.

## trajectory

2023-01 Express bare routes, no validation
2023-05 Added Zod request validation
2023-09 Trialed tRPC for 2 weeks → reverted, integration cost too high
2024-03 Added OpenAPI doc generation, current state

## rejected paths

- tRPC: 2-week trial in 2023-09; existing REST clients cost too much to migrate
  Re-evaluate when: existing clients have a defined migration path
- GraphQL: not formally evaluated; current team size and data complexity don't need it
  Re-evaluate when: frontend needs cross-resource aggregation queries

## known pitfalls

- Rate limiting not implemented: when adding it, do not break existing client retry
  logic (clients use exponential backoff)
- File uploads: auth method undefined, do not reuse existing JWT approach directly
- Error format inconsistent: legacy endpoints return { error: string }, handle
  both formats during migration

## open questions

- v2 versioning strategy not decided (URL versioning vs. header versioning)
- File upload endpoint auth design not started
```

## Layer 3: `history/*.md` — Raw Decision Events

### Purpose

`history/` stores all raw decision events with full context. It is the source of truth
for `domains/*.md` and enables precise historical queries.

### Injection Timing

On demand, when the AI needs to look up a specific past decision or rejection.

### Token Budget

**No limit.** History entries should be as detailed as needed.

### File Naming

```
YYYY-MM_<short-slug>.md
```

Examples:
- `2023-09_trpc-experiment-rejection.md`
- `2024-01_auth-debt-accepted.md`
- `2024-09_growth-stage-transition.md`

### Entry Types

| Type | When to Use |
|------|-------------|
| `decision` | A technology choice was made |
| `rejection` | A direction was excluded without adopting an alternative |
| `transition` | The approach changed from A to B |
| `debt` | A technical debt was accepted (`debt-accepted`) or resolved (`debt-resolved`) |
| `experiment` | An exploratory attempt concluded (success or failure) |

### Required Fields

```
type: <decision | rejection | transition | debt | experiment>
domain: <domain key from project domain list>
decision_date: <YYYY-MM>
recorded_date: <YYYY-MM>
summary: <one sentence>
rejected: <rejected alternatives and reasons>
reason: <why this choice was made>
revisit_when: <condition for re-evaluation>
```

| Field | Required | Notes |
|-------|----------|-------|
| `type` | MUST | One of the five types above |
| `domain` | MUST | MUST match a key in the project's locked domain list |
| `decision_date` | MUST | When the decision actually happened |
| `recorded_date` | MUST | When it was written into Cairn (may differ from `decision_date`) |
| `summary` | MUST | One sentence |
| `rejected` | MUST | **Most critical field.** Even `decision` entries MUST record what was considered and not chosen |
| `reason` | MUST | Why this path was taken |
| `revisit_when` | SHOULD | Condition under which this decision should be reconsidered |

### Dual Timestamp

`decision_date` and `recorded_date` MAY differ. A decision made in 2022 may be
recorded in 2025. The AI MUST use `decision_date` to assess time-sensitivity,
not `recorded_date`.

### The `rejected` Field

The `rejected` field is the most critical field in any history entry. It records what
was considered and not chosen — the paths the AI is most likely to re-suggest. Even
for `decision` and `transition` types, this field MUST capture alternatives that were
evaluated and discarded.

### Events That MUST Be Recorded

| Event | Entry Type | Also Update |
|-------|------------|-------------|
| Technology approach changed | `transition` | `output.md` stack + `domains/*.md` |
| Experiment failed, reverted | `experiment` or `rejection` | `output.md` no-go (if high-frequency) + `domains/*.md` |
| Known deficiency accepted | `debt` (`debt-accepted`) | `output.md` debt + `domains/*.md` |
| Technical debt resolved | `debt` (`debt-resolved`) | `output.md` debt |
| Project entered new phase | `transition` | `output.md` stage |
| Direction explicitly rejected | `rejection` | `output.md` no-go (if high-frequency) + `domains/*.md` |

### Complete Example

```
type: rejection
domain: api-layer
decision_date: 2023-09
recorded_date: 2025-01
summary: Rejected tRPC after a 2-week trial; migration cost for existing REST clients too high
rejected: tRPC — type-safe RPC layer. Two-week spike showed that migrating existing
  REST consumers (mobile app, 3 webhook integrations, 2 partner API clients) would
  require a coordinated multi-client release. No incremental adoption path found.
reason: Existing REST API surface consumed by 6+ clients. tRPC's all-or-nothing router
  model made migration a flag day, not a gradual rollout. A team of 2 could not absorb
  the coordination cost.
revisit_when: Greenfield service with no existing REST consumers, or tRPC adds
  first-class REST compatibility
```

## Naming Conventions

### Domain Keys

- Format: `kebab-case`
- Subdomains: `<domain>/<scope>` (e.g., `api-layer/user-service`, `database/primary`)
- Domain list MUST be locked at `cairn init` time
- AI MUST NOT invent new domain keys when writing history entries

### Standard Domain List

The following 11 domains are available at init time. Projects SHOULD select 3–7 that apply.

| Key | Area |
|-----|------|
| `state-management` | Frontend state management |
| `api-layer` | API design and communication |
| `database` | Data storage |
| `auth` | Authentication and authorization |
| `frontend-framework` | Frontend framework |
| `testing` | Testing strategy |
| `deployment` | Deployment and infrastructure |
| `monitoring` | Monitoring and alerting |
| `architecture` | Overall architecture patterns |
| `performance` | Performance optimization |
| `security` | Security strategy |

Custom domains are allowed. They MUST be in `kebab-case` and locked at init time.

## Concept Distinctions

Three concepts are easily confused. Their differences determine AI behavior.

| | `no-go` | `accepted debt` | `known pitfalls` |
|--|---------|-----------------|-----------------|
| **Nature** | Directional exclusion | Known defect in the codebase | Operational trap |
| **AI reaction** | Do not suggest this direction | Do not attempt to fix; work within the constraint | Actively avoid trigger conditions |
| **Exists in code?** | No (evaluation conclusion) | Yes (the defect exists) | Yes (the side-effect exists) |
| **`revisit_when`** | Optional | REQUIRED | None (persistent warning) |
| **Written in** | `output.md` + `domains/` | `output.md` + `history/` + `domains/` | `domains/` only |

**no-go** says: "we decided not to go this way."
**accepted debt** says: "we know this is broken and we're not fixing it yet."
**known pitfalls** says: "be careful when working here."

The same underlying problem CAN appear in multiple forms simultaneously. For example,
a WebSocket concurrency bug that is accepted as debt (`output.md` debt) may also generate
a known pitfall in `domains/api-layer.md` warning future work not to rely on WebSocket
consistency guarantees.

## `.cairn/SKILL.md` — Operating Protocol

`.cairn/SKILL.md` is the full Cairn operating protocol for the AI. It is **not authored
by users** — it is copied from `skills/claude-code/SKILL.md` by `cairn init` and
refreshed by `cairn init --refresh-skills`.

The protocol tells the AI:
- When and how to read each layer
- How to interpret `no-go`, `debt`, and `known pitfalls` constraints
- When to write to history, domain files, and output.md
- The FORMAT REFERENCE for history entry fields (so the AI can compose entries correctly)

AI assistants read `.cairn/SKILL.md` after reading `output.md` at session start. Users
do not edit this file. To get the latest protocol, run `cairn init --refresh-skills`.

For the normative definition of task-completion behavior and the required `Cairn reflection`
block format, see [spec/TASK-COMPLETION-PROTOCOL.md](TASK-COMPLETION-PROTOCOL.md).

## Support Directories (v0.0.11 — deprecated in v0.0.12)

> **v0.0.12 note:** `.cairn/staged/`, `.cairn/audits/`, and `.cairn/reflections/`
> are no longer created or used by Cairn. In v0.0.12, AI assistants write history
> entries and update domain files directly using their native file tools — no staging
> gate or CLI ceremony. If you have these directories from a v0.0.11 or earlier project,
> `cairn doctor` will warn you. You can review the contents and migrate useful material
> to `history/` or `domains/` manually, then delete the old directories.

`.cairn/staged/` and `.cairn/audits/` were support directories in v0.0.11. They did not
constitute additional protocol layers — they existed to maintain the official three-layer
system via a human-review staging workflow.

### `.cairn/staged/` — Candidate Inbox

Staged candidates are prefixed files pending human review. The filename prefix
determines how `cairn stage review` routes the file on accept.

#### Candidate types

| Prefix | On accept | Generated by |
|--------|-----------|--------------|
| `history-candidate_` | Move to `history/` (strip prefix) | `cairn reflect`, `cairn analyze`, `cairn log --quick` |
| `domain-update-candidate_` | Open `$EDITOR` on target `domains/<name>.md` | `cairn reflect`, `cairn analyze` |
| `output-update-candidate_` | Open `$EDITOR` on `output.md` | `cairn reflect`, `cairn analyze` |
| `audit-candidate_` | Move to `audits/` (strip prefix) | `cairn reflect`, `cairn analyze` |

#### File naming

```
<kind>-candidate_YYYY-MM_<slug>.md
```

Examples:
- `history-candidate_2026-04_trpc-rejection.md`
- `domain-update-candidate_2026-04_auth-open-question.md`
- `output-update-candidate_2026-04_stack-drift.md`
- `audit-candidate_2026-04_state-cleanup.md`

#### Backward compatibility

Files without a recognized prefix (e.g., `2024-03_foo.md` from pre-v0.0.8 tools) are
treated as `history-candidate` and moved to `history/` on accept. No rename is performed.

#### Provenance meta-comments

Files generated by `cairn analyze` carry additional meta-comment headers for provenance:

```
# cairn-analyze: v0.0.8
# confidence: high|medium|low
# source: <human-readable description>
# layer: 1|2|3
```

These comments are stripped when the file is accepted. They are orthogonal to the
candidate type prefix.

### `.cairn/audits/` — Migration Cleanup Tracking

Audit files record explicit migration cleanup obligations. They are created by
`cairn audit start` and updated by `cairn audit scan`.

#### File naming

```
YYYY-MM_<domain>-<slug>.md
```

Examples:
- `2026-04_state-management-migration.md`
- `2026-04_api-layer-cleanup.md`

#### Format

```markdown
# Audit: <description>
date: YYYY-MM
domain: <domain-key>
trigger: <what change triggered this audit>
status: open|partial|complete

## Expected removals
- <old dependency, directory, or pattern to remove>

## Findings
- <what was found: still present / confirmed removed>

## Follow-up
- <remaining action items>
```

| Field | Required | Notes |
|-------|----------|-------|
| `date` | MUST | When the audit was created |
| `domain` | MUST | Domain key from project domain list |
| `trigger` | MUST | One-line description of the change that required cleanup |
| `status` | MUST | `open` (not started), `partial` (in progress), `complete` |

#### Lifecycle

- **Create:** Run `cairn audit start <domain> --trigger "<change>"` after a significant migration
- **Scan:** Run `cairn audit scan` to populate Findings with residue detection results
- **Update:** Edit the file directly to mark items resolved; change `status` when complete
- **Archive:** Leave `status: complete` files in `audits/` for historical record; `cairn doctor` stops flagging them

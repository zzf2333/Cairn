# Cairn — 设计文档

> 状态：三层架构已定稿。

---

## 命名由来

Cairn（石堆）是登山者在岔路口和山顶堆起的一组石头。在 GPS 出现之前，登山者通过在关键位置堆石来标记进度：这条路可以走，那条山脊是死路，这里要转弯。后来的旅行者不必重新发现每一个弯路。所有前人的判断已经堆叠在路边。

这正是 Cairn 对 AI 编程助手所扮演的角色。

每次 AI 进入一个项目，它都像一个从未踏上这座山的登山者——能力出众、装备精良，但对哪些方向已被尝试过和放弃、哪些区域有隐藏危险、项目目前处于旅程的哪个阶段，一无所知。Cairn 将项目积累的决策以结构化形式堆叠在路边：这个技术方向被尝试过，没有成功；这个模块有已知陷阱，绕行；项目现在处于稳定阶段，不是大幅变更的时机。

每条拒绝记录是一块石头。每条已知陷阱是一块石头。每次阶段过渡是一块石头。石堆越完整，之后来的人——无论是新的人类开发者，还是开始新会话的 AI——就越不可能偏离方向。

## 问题所在

软件项目会积累约束。每个架构决策都由之前决策的历史所塑造：已有的技术栈、已接受的权衡、已评估并排除的方向。这种对先前选择的依赖被称为**路径依赖**。

AI 编程工具无法感知路径依赖。它们在会话之间没有记忆。每次对话从零开始——AI 可以访问当前代码，但无法获知其背后的推理。结果是一个循环往复的模式：

- AI 建议使用 tRPC。团队八个月前尝试过 tRPC，两周后回滚了。AI 不知道这件事。
- AI 建议重构认证模块。团队故意将当前的认证耦合作为技术债接受，直到团队规模扩大。AI 不知道这件事。
- AI 推荐微服务拆分。团队没有运维能力，并明确排除了这个方向。AI 不知道这件事。

这些都不是通常意义上的 AI 失败。模型在给定信息的条件下推理是正确的。问题是结构性的：它需要的信息不以 AI 可以使用的形式存在。

文档存在，但文档是为人类编写的。架构决策记录（ADR）描述做了什么决定。它们不会告诉 AI 避免提出什么建议、在什么条件下过去的拒绝值得重新考虑、或者当前的活跃约束是什么。在每次会话开始时阅读 20 页的 ADR 不是一个现实的工作流程。

结果是 AI 助手表现得像今天才加入团队、还没有完成入职培训的开发者——技术能力没问题，但不断提出早已确定的事情。每次会话都是第一天。

## 设计原则

### 约束系统，而非文档系统

Cairn 中的每条信息都必须能够改变 AI 的建议。如果删去一行不会影响 AI 的推荐，那行就不该存在。

这是比文档更严格的标准。文档可以为人类读者记录上下文、背景和理由。Cairn 只记录主动约束 AI 行为的内容：哪些方向是禁区，当前的权衡优先级是什么，AI 在哪些域上需要额外上下文才能给出有用的回答。

无法改变 AI 行为的信息——纯粹为了保存历史上下文的信息、不影响未来决策的理由——属于项目 Wiki 或 ADR，不属于 Cairn。

### 不是 ADR

ADR 和 Cairn 是互补的，而不是竞争的。它们为不同的受众服务，有不同的目标。

ADR 是为会仔细阅读、理解细微差别并应用判断的人类开发者编写的。它可以很长、叙事性强、具有探索性。它记录推理过程，为人们保存组织记忆。

Cairn 条目是为 AI 消费而构建的。它使用简短的、机器可解析的字段。它专注于 AI 作为结果应该有什么不同的行为。它必须适应 token 预算，使其在每次 AI 会话中注入上下文时是实用的。

同一个决策可以也应该同时出现在两个地方。ADR 为团队捕获完整推理。Cairn 条目为 AI 捕获行为约束。它们看起来会不同，因为它们服务于不同的目的。

### 工具无关的数据层

`.cairn/` 目录只包含带结构化文本的 Markdown 文件。它不依赖任何特定的 AI 工具、编辑器插件或平台。数据随项目在版本控制中流转，就像任何其他文件一样。

行为层——告诉每个工具如何读取 Cairn 的 Skill 文件——是特定于工具的，但只有行为规范不同。底层数据始终是相同的三层结构。项目可以从 Claude Code 切换到 Cursor，而无需迁移任何数据。同时使用多个工具的团队从同一个 `.cairn/` 目录读取。

## 为什么是三层？

构建约束系统时，第一直觉是使用单一文件。一个文件很简单：AI 读取它，人工编辑它，完成。单一文件的问题在于，不同类型的信息具有根本不同的使用模式。

全局约束需要在每次会话开始时出现，这意味着它们必须足够小，不会消耗大量上下文预算。每次会话注入一个 5000 token 的文件是不现实的。但如果为了适应 token 限制而大幅截断，就会失去使约束在规划复杂工作时有用的细节。

三层架构通过将注入时机与信息粒度匹配来解决这个问题。

### 第一层：`output.md` — 始终开启的全局约束

`output.md` 在每次 AI 会话开始时自动注入，无需任何明确请求。它回答：当前项目中适用于所有事情的不可协商约束是什么？

这一层必须通过设计保持较小（目标 500 tokens，硬限制 800）。这个约束是有意的：任何太大而无法放入这里的内容属于更深的层级。保持 `output.md` 小的纪律迫使优先级排序。如果它不断增长，那是一个信号，说明某些约束是特定域的，应该放在 `domains/` 中。

这里的信息是二元的：要么它全局适用于每个会话，要么它不属于第一层。

### 第二层：`domains/*.md` — 规划时的域上下文

域文件是关键的中间层。它们在 AI 在代码库的特定区域进行规划、架构设计或技术选型时按需注入——不是在会话开始时，也不仅仅在明确查询时。

它们解决的问题：`output.md` 可以用一行写出 `- tRPC (REST integration cost)`。这足以阻止 AI 提出 tRPC。但如果 AI 正在帮助设计一个新的 API 端点，一行是不够的。AI 需要了解 API 层决策的完整发展轨迹、尝试了什么、为什么特定替代方案被拒绝、当前的已知陷阱是什么，以及哪些设计问题仍然开放。

在规划时读取完整的 `history/` 记录也不是答案。原始历史条目是详细且未压缩的——对精确查找有用，对快速建立工作上下文无用。

域文件是预压缩的设计上下文。它们将相关历史综合成 AI 可以立即使用的形式：当前状态、发展轨迹、带重新评估条件的被拒路径、已知陷阱和开放问题。它们从积累的历史条目中一次性生成，并在项目演进时被覆盖（而不是追加）。每个文件保持在 200–400 token 范围内。它们不会随时间增长。

### 第三层：`history/*.md` — 精确查询的完整原始记录

历史文件存储每个重要决策事件的完整原始记录。它们没有 token 限制。它们不会在会话开始时或规划期间自动注入——它们在 AI 需要完整细节查找特定历史决策时按需查询。

第二层和第三层之间的区别是压缩。域文件是推导出的摘要；历史文件是源材料。域文件可能随着项目演进而过时；历史文件是只追加的且不可变的。当 AI 需要引用 2023 年 9 月 tRPC 被拒绝的确切原因，或理解技术债接受决策的完整上下文时，它直接读取历史条目。

### 三层架构的价值

组合产生了 AI 上下文质量的质的差异。考虑 AI 处理 API 层设计任务的情况：

- 来自 `output.md`：一行说明 tRPC 是禁区，一个 hook 指向域文件。
- 来自 `domains/api-layer.md`：从裸 Express 路由经过 Zod 验证到 tRPC 试验和回滚的完整发展轨迹，当前 REST + OpenAPI 状态，带重新评估条件的被拒路径，关于速率限制和错误格式不一致的已知陷阱，以及关于版本控制策略的开放问题。
- 来自 `history/`（如有需要）：tRPC 实验的完整记录——测试了什么、集成成本是什么、做出回滚决策的确切原因。

拥有这些上下文的 AI 在做 API 层设计时，拥有的信息相当于一位在这个项目工作了 18 个月的开发者。没有 Cairn，同一个 AI 只能访问当前代码，完全没有历史记录。

## `rejected` 字段是最重要的字段

Cairn 第三层中的每条历史记录都有一个 `rejected` 字段，记录了被考虑但未被选择的替代方案。它在格式规范中被标记为最关键的字段，甚至对于 `decision` 和 `transition` 条目类型也是必需的——不仅仅是 `rejection` 条目。

原因与 AI 助手的工作方式密切相关。

AI 模型被训练为有帮助。当被要求寻找解决方案时，它们会根据现有信息产生看似好的解决方案。它们无法访问已经排除某些解决方案的推理过程。结果是，AI 助手会可靠地重新提出之前考虑过并被排除的方向，因为从它们的角度来看，这些方向看起来像是合理的答案。

`rejected` 字段直接解决了这个问题。记录"我们考虑过 tRPC，以下是我们没有采用它的原因"使该推理对 AI 可用。AI 现在可以将该方向视为已经评估过的方向，而不是全新的选项。

这对于主要记录是选择的事物（而不是被拒绝的事物）的决策也很重要。当一个团队决定使用 Zustand 进行状态管理时，他们也考虑过 Redux 和 Jotai 并以特定原因排除了它们，这是 AI 需要的信息——否则它会继续将那些替代方案作为改进建议提出。`rejected` 字段使"未选择"的路径与选择的路径一样明确。

它还起到第二个作用：域文件中每条被拒路径附带的 `Re-evaluate when:` 条件给了 AI 一个关于何时应该重新审视拒绝的模型。"我们因现有客户的迁移成本而拒绝了 tRPC"暗示的重新评估条件与"我们因它不符合团队的心智模型而拒绝了 tRPC"不同。记录条件使约束是有时限的，而不是永久的。

## 采用模型：Init + 响应式

Cairn 的采用模型围绕一个核心洞察设计：维护约束系统的成本必须低于没有约束系统的成本。一个需要作为单独工作流程主动维护的系统会被放弃。Cairn 被设计为响应式维护——响应已经发生的事情更新，而不是作为主动的文档工作。

### Init：一次性历史盘点

`cairn init` 在项目采用 Cairn 时运行一次。其目标是在 30 分钟内建立初始状态——足够有用，但不期望是详尽的。

初始化过程选择项目的相关域，用当前阶段和已知约束填写 `output.md`，为团队使用的工具生成 Skill 适配文件，并创建 `domains/` 和 `history/` 的空目录结构。域文件在初始化时故意留空。它们在足够的历史条目积累后从历史条目生成，而不是从头手动编写。

初始化的指导原则是：不完整比不准确好。空的 `rejected paths` 章节比虚构的拒绝原因更有用。

### 响应式：五个触发事件

初始化之后，Cairn 只在特定事件发生时更新。模型是明确的：不要主动维护 Cairn。等待值得记录的事情发生，然后记录它。

五个触发事件是：

**A. AI 提出了你已经拒绝的方向。** 这是最常见的触发器，也是最有价值的。当 AI 建议团队已经排除的事情时，这是证据：该排除没有在 Cairn 中捕获。记录一条 `rejection` 历史条目，如果该方向可能再次出现，将其添加到 `output.md` 的 `no-go` 章节，并更新相关域文件。

**B. 做出了重大技术决策。** 团队做出重要架构或技术选择后，记录为 `decision` 或 `transition` 历史条目。如果活跃技术发生了变化，更新 `output.md` 的 `stack` 章节，并更新相关域文件。

**C. 尝试了一个方向但放弃了。** 失败的实验通常比成功的更有价值。那些不起作用的路径正是 AI 最可能建议的。记录 `experiment` 条目，重点说明为什么放弃了这种方法。更新域文件的 `rejected paths` 章节。

**D. 故意接受了一个已知缺陷。** 当团队明确决定不修复一个已知问题——通常出于优先级考虑——记录一条 `debt-accepted` 历史条目并将技术债添加到 `output.md`。这防止 AI 将修复视为明显的改进并将其纳入不相关的建议中。

**E. 项目进入了新阶段。** 阶段过渡影响所有 AI 建议的推理模式。处于稳定模式的团队不应该收到针对速度优化的建议。记录一条 `transition` 历史条目并更新 `output.md` 的 `stage` 章节。

### 为什么这个模型有效

响应式模型创造了一个正向反馈循环。当 AI 违反了尚未在 Cairn 中的约束时（触发器 A），记录它使 AI 将来不太可能违反该约束。随着时间推移，捕获的约束集增长以覆盖 AI 实际上倾向于对该特定项目错误建议的方向。

系统根据项目的实际痛点进行自我校准，而不是要求团队预先猜测应该记录什么。AI 的错误成为 Cairn 应该包含什么内容的规范。

## 工具兼容性

`.cairn/` 目录是数据层。它是纯 Markdown，与项目一起进行版本控制，不依赖任何特定工具。任何可以读取文件的 AI 助手都可以读取 Cairn。

行为层是 Skill 适配文件——每个工具一个——规定每个工具应如何与三层交互。这些文件描述注入时机（始终读取 `output.md`；当 hooks 匹配时读取域文件；按需查询历史记录），但它们读取的底层数据始终相同。

Skill 适配文件位置：

| 工具 | 位置 |
|------|------|
| Claude Code | `.claude/skills/cairn/SKILL.md` |
| Cursor | `.cursor/rules/cairn.mdc` |
| Windsurf | `.windsurfrules`（追加） |
| Cline / Roo Code | `.clinerules`（追加） |
| GitHub Copilot | `.github/copilot-instructions.md`（追加） |
| Codex CLI | `AGENTS.md`（追加） |
| Gemini CLI | `GEMINI.md`（追加） |
| OpenCode | `AGENTS.md`（追加，与 Codex 共享） |

适配文件由 `cairn init` 为团队选择的工具生成。它们可以随时重新生成，而不影响 `.cairn/` 数据目录。

所有适配文件的行为契约是相同的：在每次会话开始时读取 `output.md`；当用户请求与 hook 关键词匹配时读取相关域文件；当需要精确查找特定历史决策时查询历史条目。适配文件的格式因工具而异；语义不变。

## 路线图

Cairn 分三个阶段开发。

**Phase 1 — 协议层。** 已完成。交付物是格式规范（`spec/FORMAT.md`）、本设计文档、五个支持工具的 Skill 适配文件、一个涵盖所有三层且有 18+ 个月历史的真实项目示例，以及一个交互式 `cairn-init.sh` Shell 脚本。成功标准：三人或以上的团队能够独立运行 `cairn init` 并在不需要额外说明的情况下获得可工作的设置；记录的 `no-go` 条目可靠地防止 AI 重新建议这些方向；域文件上下文在规划期间明显提高了 AI 建议的质量。

**Phase 2 — CLI。** 已完成。带四个子命令的 `cairn` 命令行工具：`cairn init`（交互式初始化）、`cairn status`（三层摘要）、`cairn log`（手动追加历史条目）和 `cairn sync`（AI 辅助：从积累的历史条目重新生成域文件，并经人工确认）。`sync` 命令是相对于 Shell 脚本的关键新增——它在不需要手动重写的情况下关闭了原始历史积累和域文件维护之间的循环。

**Phase 3 — MCP Server。** 已完成。一个 Model Context Protocol 服务器（`mcp/`），通过类型化工具调用公开对三层的结构化访问：`cairn_output()` 读取 `output.md`，`cairn_domain(name)` 读取特定域文件，`cairn_query(domain, type?)` 搜索历史条目，`cairn_propose(entry)` 让 AI 将新历史条目草稿写入暂存区供人工审核，`cairn_sync_domain(name)` 从历史条目重新生成域文件，以及 `cairn_match(keywords)` 进行精确的基于 hooks 的域匹配而无需 AI 推断。MCP 层实现了更丰富的工具集成，并消除了每个 AI 工具手动处理原始文件注入的需要。详见 `mcp/README.md` 中的配置说明。

# Cairn — 设计文档

> 状态：v2.0 — 动态记忆引擎架构。

---

## 命名由来

Cairn（石堆）是登山者在岔路口和山顶堆起的一组石头。在 GPS 出现之前，登山者通过在关键位置堆石来标记进度：这条路可以走，那条山脊是死路，这里要转弯。后来的旅行者不必重新发现每一个弯路。所有前人的判断已经堆叠在路边。

这正是 Cairn 对 AI 编程助手所扮演的角色。

每次 AI 进入一个项目，它都像一个从未踏上这座山的登山者——能力出众、装备精良，但对哪些方向已被尝试过和放弃、哪些区域有隐藏危险、项目目前处于旅程的哪个阶段，一无所知。Cairn 将项目积累的决策以结构化形式堆叠在路边：这个技术方向被尝试过，没有成功；这个模块有已知陷阱，绕行；项目现在处于稳定阶段，不是大幅变更的时机。

每条拒绝记录是一块石头。每条已知陷阱是一块石头。每次阶段过渡是一块石头。石堆越完整，之后来的人——无论是新的人类开发者，还是开始新会话的 AI——就越不可能偏离方向。

## 问题所在

软件项目会积累约束。每个架构决策都由之前决策的历史所塑造：已有的技术栈、已接受的权衡、已评估并排除的方向。这种对先前选择的依赖被称为**路径依赖**。

AI 编程工具无法感知路径依赖。它们在会话之间没有记忆。每次对话从零开始——AI 可以访问当前代码，但无法获知其背后的推理。结果是一个循环往复的模式：

- AI 建议使用 tRPC。团队八个月前尝试过 tRPC，两周后回滚了。AI 不知道这件事。
- AI 建议重构认证模块。团队故意将当前的认证耦合作为技术债接受，直到团队规模扩大。AI 不知道这件事。
- AI 推荐微服务拆分。团队没有运维能力，并明确排除了这个方向。AI 不知道这件事。

这些都不是通常意义上的 AI 失败。模型在给定信息的条件下推理是正确的。问题是结构性的：它需要的信息不以 AI 可以使用的形式存在。

文档存在，但文档是为人类编写的。架构决策记录（ADR）描述做了什么决定。它们不会告诉 AI 避免提出什么建议、在什么条件下过去的拒绝值得重新考虑、或者当前的活跃约束是什么。在每次会话开始时阅读 20 页的 ADR 不是一个现实的工作流程。

结果是 AI 助手表现得像今天才加入团队、还没有完成入职培训的开发者——技术能力没问题，但不断提出早已确定的事情。每次会话都是第一天。

## 现有工具的不足

**Memory Bank（Cline 社区）** 用六个 Markdown 文件记录项目当前状态——技术栈、架构、进度。这解决了"项目是什么"的问题，但它是静态快照，没有时间轴，没有否决记录。

**Cursor Rules / CLAUDE.md / Copilot Instructions** 是分布式的系统提示，告诉 AI"怎么写代码"。但它们不知道"为什么这么写"——哪些决策背后有什么权衡，哪些替代方案已经被排除。

**ADR** 有否决记录，但它是写给人类看的正式文档，格式重，独立开发者几乎没人执行，而且从未被设计成 AI 可消费的上下文。

**所有这些方案的共同问题：它们都需要人类手动维护。** 一旦维护成本稍高，就没人写了，三个月后文件腐烂，系统等于不存在。

## 从 v1 到 v2：演进

### v1 做了什么

Cairn v1 交付了一个扎实的协议层：三层文件格式（`output.md` / `domains/` / `history/`）、四个 CLI 命令、六个 MCP 工具、八个 AI 工具的 Skill 适配。数据是 `.cairn/` 中的纯 Markdown，随代码版本控制。

### v1 的根本问题

v1 解决的是"AI 应该读哪些文件、这些文件怎么写"。但我们最初想要的是：

> 项目记忆应该**自动产生、自动演化、自动约束未来 AI 行为**。

v1 的根本差距：**用户仍然要手写 `.cairn/` 文件。** AI 只是被动读取。没有信号捕获，没有记忆准入控制，没有记忆演化，没有项目阶段推理。

v1 是**静态上下文格式**。我们需要的是**动态记忆引擎**。

### v2 的转变

v1 信息流：`人类写 → .cairn/ 文件 → AI 读取`

v2 信息流：

```
项目事件自然发生
  → 双耳捕获信号（Git 变更 + AI 对话）
  → 信任策略过滤分级
  → 结构化记忆自动生成
  → AI 可消费的约束视图自动产生
  → AI 在约束下工作
  → 新交互继续产生信号
  → 人类只在高风险变更时审核
```

用户从不手写记忆。所有记忆由 Server 自动或半自动产生。

## 双耳设计

### 为什么需要两个信号源

项目记忆所需的信号分布在两个本质不同的渠道。任何一个渠道都无法提供完整画面。

### Git 耳朵：能给"发生了什么"，不能给"为什么"

Git 耳朵观察代码层面的事实：

- **回滚** — 某个方向被尝试后撤回
- **依赖变更** — 包出现后消失，或被替换
- **大规模文件移动** — 重大重构事件
- **提交频率趋势** — 加速、减速、稳定
- **新贡献者** — 团队组成变化

这些是客观的项目演化事实。但 Git 无法告诉你*为什么*回滚、*为什么*移除依赖、*为什么*重构。行为背后的推理对 Git 是不可见的。

### 对话耳朵：能给"为什么"，不能给"发生了什么"

对话耳朵捕获用户在 AI 交互中说的话：

- **用户否决** — "别建议那个，我们试过了"
- **历史引用** — "我们之前做过这个，失败了"
- **约束声明** — "现在在冲刺融资，别搞大改"
- **决策** — "状态管理我们用 Zustand"
- **债务接受** — "认证耦合我们知道，先接受"

这些是最高价值的约束信号，但只存在于人和 AI 的对话中。Git 里拿不到。

### 两者结合

只有 Git 耳朵，知道发生了什么，不知道为什么。
只有对话耳朵，知道用户说了什么，不知道代码实际怎么变的。

两只耳朵合起来才能完整捕获路径依赖：Git 耳朵提供事实基底，对话耳朵提供推理层，组合产生同时携带证据和意图的记忆条目。

## 信任策略：为什么不能完全自动

### 核心论点：记忆主权

完全自动写入记忆是危险的。一条错误的 no-go 会像错误的系统提示词一样不断放大——AI 后续所有建议都会被这条错误约束影响。

学术界对此有明确警告（arXiv 2604.16548，Mnemonic Sovereignty）：**危险不仅在于插入单条错误条目，而在于该错误通过总结、反思或经验记忆机制被提升为高权重。**

Cairn 的立场：**任何会改变 AI 全局行为的记忆，永远不能完全自动写入。**

### L0–L3 四级信任

| 级别 | 名称 | 存储位置 | 行为 |
|------|------|----------|------|
| L0 | Drop | 不存储 | 噪声、重复、低置信度，丢弃 |
| L1 | Candidate | `signals/` | 积累等待佐证 |
| L2 | Staged | `staged/` | 等待人工审核 |
| L3 | Auto-write | `memory/` | 严格条件下自动写入 |

### 硬规则：永远不自动

以下条件始终路由到 L2，不论来源可靠度或置信度：

- 新增全局 no-go
- 阶段变更
- `scope == global` 的任何 `behavior_effect`
- `never_auto` 配置列表中的任何条目

### 信号如何在级别间流转

```
信号进入 Trust Router
  │
  ├→ 重复？→ 合并到已有条目 → L0/merge
  │
  ├→ 触发硬规则 L2？→ staged/（不论来源多可靠）
  │
  ├→ 满足 L2 条件？→ staged/
  │
  ├→ 满足 L3 条件？→ memory/ → 触发 views 重新生成
  │
  ├→ 置信度 >= medium → L1（signals/）
  │
  └→ 置信度 < medium → L0（丢弃）
```

同一 domain + 同一 subject 的 L1 信号积累超过阈值时，自动升级为 L2。

### L3 条件：何时允许自动写入

自动写入需要同时满足：来源可验证（如 git-revert）、范围为 local、与现有记忆无冲突、不在 `never_auto` 列表中。

例：检测到本地依赖的 Git 回滚，来源客观（Git 证据）、范围局部、无冲突——可以作为 L3 自动写入。

反例：用户说"我们永远不用 GraphQL"，这影响全局行为，无论置信度多高，必须走 L2。

## 阶段顾问引擎

### 独有能力

在所有调研到的工具和论文中，**没有任何一个做过从项目信号自动推断项目阶段。** 同一个技术建议在探索期和成熟期的答案完全不同，但现有 AI 工具都不知道项目处于哪个阶段。

### 规则版 v0.1

阶段引擎综合多维信号：项目年龄、提交频率趋势、依赖变更率、新文件比例。

### 四个阶段

| 阶段 | 特征 | AI 引导 |
|------|------|---------|
| **探索期** | 年轻项目，依赖变动大 | 偏向速度，容忍实验 |
| **增长期** | 提交增多，依赖趋稳 | 平衡速度与稳定 |
| **成熟期** | 新文件少，架构稳定 | 新依赖需要充分理由 |
| **维护期** | 提交频率下降，变动小 | 优先稳定，最小化风险 |

### 仅作咨询

阶段引擎输出严格为 advisory。置信度低于阈值时不产生硬约束。阶段判断只有经过人工确认后才成为正式约束。阶段变更永远走 L2。

## Memory / Views 分离

### 问题

v1 中 `output.md` 和 `domains/*.md` 既是源数据又是运行时上下文——不知道哪些内容是原始事实、哪些是摘要。AI 修改摘要时可能破坏历史真实性，Git diff 无法区分事实变化和投影变化。

### v2 数据模型

**`memory/` 是源数据。** 每条记忆是 YAML 文件，包含完整来源追溯、置信度、`behavior_effect` 声明。这是人类审核的对象。

**`views/` 是自动生成的视图。** `output.md`、`domains/*.md`、`stage.md` 全部从 `memory/` 自动聚合生成。AI 消费视图，人类审核源数据。

### 分离的好处

**防止记忆腐败。** views 随时可从 memory 重新生成，源数据永远不是渲染后的视图。

**Git diff 清晰度。** YAML 结构化 diff 比 Markdown 散文变更更容易审阅。审核者可以看到具体哪个字段变了（`confidence`、`behavior_effect`、`revisit.status`）。

**v1 兼容降级。** `views/` 格式和 v1 的 `output.md` / `domains/` 完全兼容。v1 的 Skill 适配文件可以直接读取 `views/` 作为降级方案。

### 生成规则

- `views/output.md` — 全局 no-go、活跃技术栈决策、已接受债务、域 hooks、阶段顾问。目标 500 tokens，硬限制 800。
- `views/domains/*.md` — 按域分组记忆条目，生成轨迹、被拒路径、已知陷阱、开放问题。每个文件目标 300 tokens，硬限制 500。
- `views/stage.md` — 当前阶段判断详情。

所有 views 文件带生成头部标注：

```markdown
<!--
Generated by Cairn. Do not edit manually.
Source: .cairn/memory/*.yaml
-->
```

## 设计原则

### 1. 不删模块，降深度

v2 从第一天就交付所有核心模块——Git 耳朵、对话耳朵、信任策略、阶段引擎、记忆引擎、视图引擎。但每个模块的智能深度在 v0.1 用规则实现，不用 LLM。架构第一天完整，智能逐版本加深。

### 2. 不降架构，控权限

系统骨架完整，但 L3 自动写入条件严格。默认姿态保守：有疑问就路由到 L2 人工审核。随着系统在实际使用中证明可靠，逐步放宽 L3 条件。

### 3. 人审核权不可绕过

任何改变 AI 全局行为的记忆——全局 no-go、阶段变更、全局 `behavior_effect`——必须经过 L2 人工审核。无论配置变更、置信度积累还是自动化流程，都不能绕过这道门。

### 4. 事实与推测分离

`cairn_plan` 是纯只读顾问工具，永远不得写入 `signals/`、`staged/` 或 `memory/`。AI 的规划推测不等于项目事实记忆。

### 5. 渐进信任，不跳级

信号从 L1 开始积累，升级到 L2 等待审核，经审核或满足 L3 条件后才进入 `memory/`。没有从原始信号一步跳到永久记忆的机制。

## `rejected` 字段仍是最重要的字段

每条记忆都带有 `rejected` 字段，记录被考虑但未被选择的替代方案。v1 中这是最关键的字段，v2 中依然如此。

AI 会可靠地重新提出之前考虑过并被排除的方向——因为从它的角度看，这些方向是合理的答案。`rejected` 字段直接解决这个问题。`revisit.when` 条件让约束有时限而非永久。

v2 中，`rejected` 字段获得了更多结构：它存在于 YAML 源数据中，带有显式 `behavior_effect` 声明，使其约束角色可被机器验证，而不依赖 AI 对散文的解读。

## 采用模型：从响应式到持续捕获

### v1：init + 五个触发事件

v1 的响应式模型是合理的：运行一次 `cairn init`，然后在特定事件发生时响应式更新。

### v2：持续捕获

v2 保留响应式模型但自动化了捕获步骤。不再需要用户发现触发事件并手动记录，双耳系统持续检测信号：

- Git 耳朵在每次 Server 启动时扫描回滚、依赖变更、结构性变化
- 对话耳朵在自然 AI 交互中捕获否决信号、约束声明、决策

用户角色从"写记忆"变为"审核记忆"。

## 工具兼容性

Cairn v2 以 stdio MCP Server 运行。支持 MCP 协议的 AI 工具可通过类型化工具调用原生使用：`cairn_context()`、`cairn_signal()`、`cairn_session_end()`、`cairn_status()`、`cairn_plan()`、`cairn_doctor()`。

不支持 MCP 的工具通过 `views/` 目录降级。`views/` 保持 v1 格式兼容，v1 Skill 适配文件可直接读取。

| 工具 | MCP 支持 | 降级方案 |
|------|----------|----------|
| Claude Code | 原生 MCP | views/ via CLAUDE.md |
| Cursor | 原生 MCP | views/ via cairn.mdc |
| Cline / Roo Code | 原生 MCP | views/ via .clinerules |
| Windsurf | 原生 MCP | views/ via .windsurfrules |
| GitHub Copilot | — | views/ via copilot-instructions.md |
| Codex CLI | — | views/ via AGENTS.md |
| Gemini CLI | — | views/ via GEMINI.md |

运行模式：AI 工具打开项目时启动，关闭时退出。每个项目独立实例，零全局 daemon，零数据库，零云端依赖。和语言服务器（LSP）完全一样——项目级、会话级。

## 路线图

**Phase 1 — 协议层（v1，已完成）。** 格式规范、八个工具的 Skill 适配、CLI、MCP Server。确立 `.cairn/` 作为数据层。

**Phase 2 — 动态记忆引擎（v2，进行中）。** 核心交付：双耳信号捕获、L0–L3 信任策略、YAML 记忆存储、自动视图生成、规则版阶段引擎、六个 MCP 工具、两步式 init、`cairn review` 人工审核。v0.1 交付所有模块（规则版智能），后续版本加深每个模块。

**后续版本：**
- **v0.2** — 信号质量增强
- **v0.3** — 阶段引擎增强
- **v0.4** — 规划引擎增强
- **v0.5** — 记忆维护增强
- **v1.0** — 自主项目记忆：长期运行验证，可量化的 AI 改善，稳定 API
